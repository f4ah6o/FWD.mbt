# 2026-03-10 開発日誌

## 今日の計画

1. v6 workbench で実用的な業務ケースを扱えるようにする。
2. 足りない機能があれば、simulation 用に追加実装する。
3. 実行例を fixture とドキュメントで固定する。

## 実装したこと

- `src/workbench_v6/` に practical preset を追加した。
  - `purchase_request_builder()`
  - `builder_for_case("purchase-request")`
- `src/ui/workbench_v6/` と `src/api_v6/` を更新し、
  - hidden `state_slots`
  - hidden `transition_slots`
  を使って state / transition slot 数を可変で往復できるようにした。
- `GET /v6/workbench?case=purchase-request` を追加した。
- practical fixture を追加した。
  - `examples/workbench_v6/purchase_request_expected.html`

## 実用ケース

今回の practical case は **備品購入申請** を想定している。

- states
  - `Draft`
  - `Submitted`
  - `ManagerApproved`
  - `FinanceApproved`
  - `Ordered`
  - `Rejected`
- transitions
  - `submit`
  - `manager_approve`
  - `manager_reject`
  - `finance_approve`
  - `finance_reject`
  - `place_order`

preview 上では次のように available transition が見える。

- `Draft`
  - `submit -> Submitted`
- `Submitted`
  - `manager_reject -> Rejected`
  - `manager_approve -> ManagerApproved`
- `ManagerApproved`
  - `finance_reject -> Rejected`
  - `finance_approve -> FinanceApproved`
- `FinanceApproved`
  - `place_order -> Ordered`
- `Ordered`
  - available: none
- `Rejected`
  - available: none

## 学び・メモ

- first slice の 3 slot 固定は demo としては十分でも、実用ケースの simulation には不足していた。
- hidden field で slot 数を round-trip させると、UI と parser を崩さずに practical case へ広げやすい。
- preset case を持つことで、authoring-first な surface のままでも業務設計の説明がかなり具体化できる。
