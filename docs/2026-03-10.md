## 追加実装: 設計を保ったまま実運用へ出す導線

「FWD の design / 将来構想を崩さずに実運用を始めたい」という目的に対して、
**schema lifecycle** と **rollout mode** を workbench に持ち込んだ。

- schema lifecycle
  - `Draft`
  - `Reviewing`
  - `Released`
  - `Deprecated`
- rollout mode
  - `simulation`
  - `shadow`
  - `assisted`
  - `enforced`

これは、

- 設計上は「今このスキーマはどの governance 状態か」
- 運用上は「今このスキーマを production にどこまで接続してよいか」

を分けて表現するための追加である。

workbench には rollout section を追加し、

- readiness
- headline
- gate
- action
- surface

を deterministic に描画するようにした。

これにより、v6 は単なる schema playground ではなく、
**design-first workbench から shadow/assisted/enforced rollout へ進む入口**
として振る舞えるようになった。

## 実用ケース

今回の practical case は **備品購入申請** を想定している。

- states
  - `Draft`
  - `Submitted`
  - `ManagerApproved`
  - `FinanceApproved`
  - `Ordered`
  - `Rejected`
- transitions
  - `submit`
  - `manager_approve`
  - `manager_reject`
  - `finance_approve`
  - `finance_reject`
  - `place_order`

この preset では、実運用の始め方を示すために

- `schemaStatus = Released`
- `rolloutMode = shadow`

を初期値にしている。

つまり「設計は一度 release されたが、まだ system of record にはせず、
shadow で traffic を mirror しながら drift を観測する」という位置づけである。

preview 上では次のように available transition が見える。

- `Draft`
  - `submit -> Submitted`
- `Submitted`
  - `manager_reject -> Rejected`
  - `manager_approve -> ManagerApproved`
- `ManagerApproved`
  - `finance_reject -> Rejected`
  - `finance_approve -> FinanceApproved`
- `FinanceApproved`
  - `place_order -> Ordered`
- `Ordered`
  - available: none
- `Rejected`
  - available: none

## 学び・メモ

- first slice の 3 slot 固定は demo としては十分でも、実用ケースの simulation には不足していた。
- hidden field で slot 数を round-trip させると、UI と parser を崩さずに practical case へ広げやすい。
- preset case を持つことで、authoring-first な surface のままでも業務設計の説明がかなり具体化できる。
- 「設計の状態」と「運用の状態」は別軸として持つ必要がある。これを一緒くたにすると、
  まだ review 中なのに production へ出してしまう、あるいは released なのに simulation から進めない、
  という判断が UI 上で曖昧になる。
- `Released + shadow` は、FWD の北極星を崩さずに現実の導入を始める上でかなり良い中間点だった。
